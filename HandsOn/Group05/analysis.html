Updated Schema post-data cleaning

When considered the state of the previous datasets, several issues were present.
The main challenge was that one row in the file did not correspond to one observation, but rather several.
In addition, the data spanned over 2 files and the column names changed depending on the unit being measured.
We have fixed this issue by creating just a single column for the primary observation, as well as a column for the background observation.
Now just one file is present. This process is known as wide-to-long reshaping in data cleaning.
Furthermore, there was no way to link data concretely with just the columns of the street names where the observations were captured.
We added 2 colummns that identify the precise latitude and longitude of where the measurement was captured.
This will make it possible to link to other data corresponding to the same city/county for Riverside CA.
Lastly, we have eliminated many columns pertaining to date because they are unnecessary for our purposes; everything in those columns can be inferred from the timestamp.

Resource Naming Strategy

The resource naming strategy:
http://riversidepollutionstudy.org/
We will use the slash "/" to define our resources and our properties in our ontology because our dataset is large and we want to be able to provide individual mesaurements
To differentiate between classes, properties, and instances, we will use the following strategy
For general classes, we will define classes with "/". An example would be http://riversidepollutionstudy.org/Observation
For properties, we will define them with "/property". An example would be http://riversidepollutionstudy.org/hasPart
For instances, we will define them with the general scheme: "/CLASS_NAME/instance/id" an example would be: "http://riversidepollutionstudy.org/Dataset/instance/1"
